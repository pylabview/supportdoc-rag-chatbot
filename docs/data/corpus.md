# Corpus Specification

## Purpose

This document defines the **primary documentation corpus**, **snapshot strategy**, **license compliance plan**, and **ingestion scope rules** for the **SupportDoc RAG Chatbot** project.

It is the human-readable source of truth for:

- what documentation is allowed into the system,
- how the corpus is versioned for reproducibility,
- how attribution is preserved for citations and exports, and
- which related artifacts are expected from the ingestion pipeline.

---

## Primary Corpus

- **Corpus name:** Kubernetes Documentation
- **Primary site:** https://kubernetes.io/docs/
- **Project use case:** technical support-style question answering over product documentation

This corpus was selected because it is open, well-structured, and suitable for retrieval-augmented generation (RAG) with evidence-based answers and clickable citations.

---

## Snapshot Strategy

The corpus is built from a **pinned snapshot** of the Kubernetes documentation repository.

- **Snapshot method:** Git commit hash
- **Why this method:** a pinned commit is the most reproducible option and avoids corpus drift caused by ingesting a live website that changes over time

### Snapshot Metadata

> Replace the placeholder values below before the first production ingestion run.

- **Source repository:** https://github.com/kubernetes/website
- **Snapshot ID:** `<snapshot_commit_hash>`
- **Snapshot date:** `<YYYY-MM-DD>`
- **Suggested snapshot label:** `k8s-<short_commit_hash>`
- **Source base URL:** https://kubernetes.io

### Reproducibility Rule

All ingestion runs, retrieval experiments, evaluations, and demo deployments must reference the same pinned snapshot ID unless an intentional corpus upgrade is documented.

## Allowlisted Source Base URLs

The ingestion pipeline is allowed to process content from the following public documentation areas:

- https://kubernetes.io/docs/
- https://kubernetes.io/docs/concepts/
- https://kubernetes.io/docs/tasks/
- https://kubernetes.io/docs/tutorials/
- https://kubernetes.io/docs/reference/ *(optional for v1; recommended if needed for command/reference coverage)*

### Repo-Relative Allowlist Prefixes

When ingesting from the `kubernetes/website` repository, the pipeline should allow only the following prefixes:

- `content/en/docs/concepts`
- `content/en/docs/tasks`
- `content/en/docs/tutorials`
- `content/en/docs/reference`

### Repo-Relative Denylist Prefixes

The pipeline should exclude the following prefixes by default:

- `content/en/blog`
- `content/en/community`
- `content/en/case-studies`

### Explicit Exclusions

The following content is intentionally excluded from ingestion unless a later architecture decision says otherwise:

- blog posts
- community pages
- GitHub issues, pull requests, and discussions
- non-English documentation
- images, binaries, and static assets
- autogenerated navigation menus and other page chrome
- any content outside the approved documentation snapshot

### Scope Rule

If a source is not explicitly allowlisted, it is **out of scope** for ingestion.

---

## Supported File Types

The initial ingestion pipeline should process only:

- `.md`
- `.mdx`

Other file types are excluded unless explicitly added to the ingestion policy.

## License and Attribution Plan

Kubernetes documentation is licensed under:

- **License:** Creative Commons Attribution 4.0 International (CC BY 4.0)
- **License URL:** https://creativecommons.org/licenses/by/4.0/

### Attribution Requirement

Every chunk stored by the retrieval system must preserve attribution metadata so the UI and exported artifacts can render compliant citations.

### Required Per-Chunk Metadata

Each chunk should include at minimum:

- `source_url`
- `license`
- `attribution`
- `snapshot_id`
- `doc_id`
- `chunk_id`
- `section_path`
- `start_offset`
- `end_offset`
- `token_count`

### Standard Attribution String

Use the following attribution string unless the upstream documentation changes its attribution requirements:

`Kubernetes Documentation © The Kubernetes Authors — CC BY 4.0`

### UI / Export Expectation

Citations should render the source URL and attribution in the UI and preserve the same attribution in evaluation outputs or exported evidence artifacts.

## Related Implementation Artifacts

This document explains the corpus rules for humans. The following artifacts are expected to implement and enforce those rules:

- **Manifest builder:** `src/supportdoc_rag_chatbot/ingestion/build_manifest.py`
- **Generated manifest:** `data/manifests/source_manifest.jsonl`
- **Runtime config:** `src/supportdoc_rag_chatbot/resources/default_config.yaml` *(recommended location for corpus config)*

### Responsibility Split

- `docs/data/corpus.md` → human-readable corpus contract
- `default_config.yaml` → machine-readable corpus settings
- `build_manifest.py` → enforcement of allowlist/denylist rules
- `source_manifest.jsonl` → audit trail of approved source files

## Rebuild Instructions

A typical rebuild flow should look like this:

```bash
python -m supportdoc_rag_chatbot.ingestion.fetch_snapshot
python -m supportdoc_rag_chatbot.ingestion.build_manifest
python -m supportdoc_rag_chatbot.ingestion.parse_docs
python -m supportdoc_rag_chatbot.ingestion.chunk_docs
python -m supportdoc_rag_chatbot.ingestion.validate_corpus
```

If some commands do not exist yet, this section should still be kept as the target ingestion contract.
